Job Consulting Platform Documentation

Phase 1: Architecture & Integration

Project Architecture

The system utilizes a Layered Architecture to separate web concerns (Controllers), business logic (Services), and data structures (Models).

Design Patterns Implemented

Strategy Pattern (UC5): Encapsulates payment logic for Credit Cards, PayPal, and Bank Transfers.
Observer Pattern (UC7): Decouples payment execution from transaction history logging.
Factory Pattern: Centralizes the creation of payment strategies based on user input.

Cross-Module Collaboration

The platform's architecture is designed for seamless interaction between subsystems. The Payment Engine does not operate in a vacuum; it actively consumes data from the Administrative Module and the Consultant Subsystem to enforce business rules.

Regulatory Compliance (UC12): The PaymentService references the SystemPolicy model to dynamically adjust platform fees and cancellation windows defined by the Admin.
Stakeholder Verification (UC11): Every transaction validates the Consultant and their RegistrationStatus, ensuring that funds are only processed for approved.

Implementation Highlights

  • Simulation Accuracy: Per Section 5.1.2, the PaymentService implements a 3000ms thread sleep to simulate gateway latency.
  • Data Traceability: The PaymentRecord model ensures every successful transaction is linked to a clientId and a unique transactionId.
  • Admin Oversight: The Consultant model utilizes a RegistrationStatus enum, enabling the Admin (UC11) to control platform access.
Validation Engine: Each strategy implements unique validation logic. The PayPalStrategy utilizes Regular Expressions (Regex) to verify account identifiers, while CreditCardStrategy performs length and format checks on CVV and Expiry fields.
